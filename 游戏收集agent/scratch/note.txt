给自己的话：在写的时候，把流程做成一个完全是机械的步骤，不见得是好事。为了节省token，或者切换低能力模型去定死流程也不见得是聪明的事情。
尽管在某些地方，我们需要约束和要求，然而还是应当把Agent当做是一个很聪明的人，再给定合适的原则时，他有能力自己去产生判断，找到方法，完成工作，并在同一个session的上下文产生记忆。哪些东西应该被固定下来？哪些东西不应该？这是一个复杂的问题。
马东锡的文章最后一句写：Anything that must be reliable belongs in the harness. CLI，流程事实上也都是harness或者Instruction，当然不见得是好的harness。

antirez的文章其实也是类似的意思，在写测试的时候，不仅可以通过代码的测试去写，也可以给agent提供一些需要测试的重点，他是足够聪明的，他是知道如何去完成任务的。如果在之后某些测试变得必要，那么我再把它通过代码固定下来。
```
传统上，软件测试使用测试套件，该套件由局部范围测试和集成测试组成（例如 Redis：测试 `SET foo 10` 是否与 `GET foo => 10` 匹配，以及在这种情况下复制功能是否正常工作）。然后，通常需要手动执行质量保证 (QA) 测试，以发现可运行测试套件中的漏洞。众所周知，覆盖所有代码行并不意味着覆盖所有可能的状态。此外，集成测试在结构上也存在诸多困难：存在许多时间安排问题、设置问题以及某些质量输出只能通过人工检查而无法自动检查，这些因素导致许多测试机会因时间或后勤限制而无法真正利用。
LLM（逻辑逻辑模型）在现有测试方法的基础上提供了一种新的质量保证（QA）方式。其核心思想是创建一个 Markdown 文件，让 AI 代理扮演 QA 工程师的角色，对新版本执行一系列手动测试。例如，对于 DwarfStar（一个用于开放权重 LLM 的推理引擎），我采用了以下方法。在 Markdown 文件中，代理需要检查软件项目已发布版本上的所有新提交。然后，模型会收到一份需要执行的操作列表，例如：
- 检查 MacBook A 和 MacBook B 上的分布式推理是否正常工作，确保输出一致，并且推理能够处理两台机器上所有 GGUF 文件，……
- 请确保此版本不包含任何速度下降的问题。
诸如此类。值得注意的是，在速度回归测试部分，我无需告知代理之前的预期速度，因为这是一个会随着新版本和优化而变化的动态目标。同样，分布式推理的集成测试也不需要太多指令，文件开头只有 SSH 端点和要使用的密钥、路径等等。
代理需要检查长长的 QA 活动列表，尤其要考虑到新增的提交，首先要检查变更并确定哪些内容可能会受到影响，以便 QA 测试能够更有针对性地查找具体的回归问题。
```
